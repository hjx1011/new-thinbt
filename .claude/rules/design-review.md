# 设计约束

## ⚠️ 强制规则

**改代码前必须先用 Read 工具读取对应文档。跳过这一步 = 架构错误。这是已验证的问题，不是建议。**

- 涉及 `src/seed/`、`src/daemon/`、`src/cdc/` → 第一个 tool call 必须是 `Read(design.md)`
- 涉及任何 bug 修复 → 第一个 tool call 必须是 `Read(BUG_ANALYSIS.md)`
- 读完文档才开始改代码，不能在读文档之前执行任何 Edit/Write

## 修改前必读文档
- 涉及种子文件 → 先读 `design.md`，理解 .tseed v3/v4 向后兼容
- 涉及 I/O 层 → 考虑机械硬盘（HDD）性能和碎片问题，参考分段方案
- 涉及传输协议 → chunk 不可分割，peer 和 tracker 不感知 segment

## 架构分层
- P2P 协议层、调度层、I/O 层严格分层，下层对上层透明
- 不要跨层调用，不要在协议层直接操作文件

## 关键约束
- 教室千兆交换机，教师机只有 1 个千兆上行端口，60 台学生机共享
- 学生机可能是机械硬盘，随机写性能极差
- 一次只打开一个段文件（最多 1 个 fd）
- 段合并是幂等的（覆盖写入），崩溃安全

## 改动影响范围检查
- 新增结构体 → 检查是否需要更新 .tseed 格式版本
- 新增公开函数 → 确认前缀、文档注释、调用者释放内存
- 修改分块逻辑 → 所有 chunk hash 会变，种子需重新生成
