# 方案先行规则

## 核心原则

**先思考，再动手。先方案，再代码。**

## 触发条件

收到任何**非平凡修改请求**时，在写第一行代码之前，必须先：

1. 分析需求，理解改动范围
2. 列出具体方案（改哪些文件、怎么改、为什么这样改）
3. 等用户确认后再开始写代码

## 什么算"非平凡"（必须先出方案）

- 新增函数/结构体/模块
- 修改 2 个以上的文件
- 涉及架构或协议变更（.tseed 格式、P2P 消息结构、分层边界）
- 性能优化（改变算法或数据结构）
- 重构已有代码
- 任何用户说"优化一下"、"改改"但没有说怎么改的场景
- 不确定用户意图时（宁可多问，不要猜）

## 什么可以直接改（不需要方案）

- 修改变量名、函数名（重命名）
- 修改单行 bug（语法错误、类型错误）
- 加注释
- 修改单文件、单函数且用户给出了精确指示

## 方案格式

```
## 方案

### 改什么
- 文件A: 改动内容
- 文件B: 改动内容

### 为什么
- 原因1
- 原因2

### 风险
- 可能影响的模块/功能
```

## 原则

宁可多问一句，也不要自作主张。用户说"改"的时候可能没想清楚，你的职责是先帮他想清楚。
